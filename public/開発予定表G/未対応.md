```

- println! は一切使わない。全てログ関数で出力する。
- ログファイルの名前: info.log
- ログファイルの配置: exeと同じディレクトリ（レジストリのInstallLocationから取得）

- ファイルが存在しない場合は自動生成する。
```

## 指示1: ログ共通関数の作成と println! の置き換え

### Step 1: 各アプリの現状調査

以下のキーワードで全Rustファイルを検索し、ヒット箇所を列挙すること。

```
println!
eprintln!
print!
```

ヒットした箇所ごとに以下を報告する：
- ファイルパス・行番号・コード抜粋
- 文脈から判断したログレベル（WARN / ERROR）

### Step 2: 共通ログ関数の実装

各アプリの `main.rs` または `utils.rs` に以下の共通関数を追加する。
`install_dir()` はレジストリから取得する既存関数を使用すること。

```rust
/// WARN / ERROR のみ記録する。INFOは記録しない。
/// ファイルが存在しない場合は自動生成する。
fn log_msg(level: &str, msg: &str) {
    if let Some(path) = install_dir().map(|p| p.join("info.log")) {
        if let Ok(mut f) = OpenOptions::new().create(true).append(true).open(&path) {
            let now = Local::now().format("%Y-%m-%d %H:%M:%S");
            let _ = writeln!(f, "[{}] [{}] {}", now, level, msg);
        }
    }
}

fn log_warn(msg: &str) { log_msg("WARN",  msg); }
fn log_err (msg: &str) { log_msg("ERROR", msg); }
```

### Step 3: println! の置き換え

Step 1で列挙した全箇所を以下のルールで置き換える。

- `println!(...)` → 文脈がエラーなら `log_err(...)` 、警告なら `log_warn(...)`
- `eprintln!(...)` → `log_err(...)`

置き換え後、`println!` / `eprintln!` の残存がないことをgrepで確認すること。

### 注意
- Polarisはすでに `log_warn` / `log_err` 関数を持っている。ログファイル名を `error_info.log` から `info.log` に変更し、パスを `install_dir().join("info.log")` に修正すること。
- 既存の `error_info.log` / `crash.log` 等の旧ファイル名参照を全て `info.log` に統一すること。

## 指示2: NSIS taskkill の非表示・エラー抑制

### 対象ファイル
- `StellaRecord/stella_record_ui/src-tauri/windows/hooks.nsi`
- `Alpheratz/src-tauri/windows/hooks.nsi`

### 修正内容

Polarisの `hooks.nsi` の `taskkill` 記述を参照し、全く同じ方式に揃えること。
具体的には `nsExec::Exec` を使用し、ウィンドウ非表示かつエラー出力を抑制した形にする。

```nsis

ExecWait 'taskkill /F /IM stellarecord.exe'

; 修正後（Polarisと同じ方式）
nsExec::Exec 'taskkill /F /IM stellarecord.exe 2>nul'
```

Polarisの実装と完全に一致させること。独自の書き方にしないこと。

#### 6-2: 全アプリの Capabilities でネットワーク通信を明示的に停止

- `network:*`
- `http:*`
- その他外部通信に関するもの

#### 6-3: ログファイル名の統一（全アプリ）

全アプリで `error_info.log` / `crash.log` 等の旧ログファイル名を参照している箇所を `info.log` に変更する。通常メッセージ・警告・エラーの全てを `info.log` 1ファイルに書き込む設計のため、用途別にファイルを分けないこと。

ヒットした箇所を全て列挙し、`info.log` に差し替えること。勝手に削除・統廃合は行わず、必ずファイル名の差し替えのみ行うこと。

## 指示9: Vite 設定の sync スタイル統一

### 対象ファイル
- `StellaRecord/stella_record_ui/vite.config.ts`
- `Alpheratz/vite.config.ts`
- その他 `vite.config.ts` が存在する箇所全て

### 修正内容

`defineConfig` に async 関数を渡している箇所を sync スタイルに統一する。

```typescript
// 修正前（asyncスタイル）
export default defineConfig(async () => {
  return {
    plugins: [react()]
  }
})

// 修正後（syncスタイル）
export default defineConfig({
  plugins: [react()]
})
```

async が不要な理由がある場合（await を実際に使用している場合）は修正せず「要確認」として報告すること。

## 指示11: 環境変数取得の脆弱なフォールバック修正

### 対象
全アプリの全Rustファイル

### 修正内容

以下のキーワードで全ファイルを検索し、ヒット箇所を全て列挙してから修正する。

検索キーワード: `unwrap_or_default` / `unwrap_or_else` / `unwrap_or("")`

環境変数（`LOCALAPPDATA` / `APPDATA` / `USERPROFILE` 等）の取得失敗時に空文字やカレントディレクトリにフォールバックしている箇所を、失敗時にNoneを返す形に変更する。

```rust
// 修正前（危険: 失敗時に空文字で動作継続）
let p = var("LOCALAPPDATA").unwrap_or_default();

// 修正後（安全: 失敗時はNoneを返して処理を中断）
let p = var("LOCALAPPDATA").ok()?;
```

呼び出し側でNoneを受け取った場合はlog_errでエラーを記録して処理を中断すること。ただし、環境変数以外の用途で `unwrap_or_default` を使っている箇所は修正しないこと。

## 指示12: pleiades.json と jewelBox.json を plugin.json に統合

### 前提条件

この指示は指示8-Bの調査・報告が完了してから実施すること。報告を受ける前に着手しないこと。

### 修正内容

`pleiades.json` と `jewelBox.json` を1つの `plugin.json` に統合する。

作業手順：

1. 指示8-Bで報告された両ファイルの内容・構造を確認する
2. `plugin.json` として統合したJSONの構造案を作成し、実施前に報告すること。勝手に統合を開始しないこと
3. 承認を得てから、StellaRecord・Alpheratz両方のコードで `pleiades.json` / `jewelBox.json` を参照している箇所を全て `plugin.json` に差し替える
4. 旧ファイル（`pleiades.json` / `jewelBox.json`）の扱いは削除せず残しておくこと。削除は別途指示する

### 対象外

UIの変更は今回対象外とする。画面表示・コンポーネントには一切触れないこと。バックエンドのファイル参照パスの差し替えのみ行うこと。
