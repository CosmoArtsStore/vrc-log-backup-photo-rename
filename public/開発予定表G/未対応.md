## プロジェクト全体の構造理解（前提知識）

このリポジトリは以下の4アプリで構成されるエコシステムである。

```
【Layer 1 / インフラ層】
Polaris        VRChatのログファイルをリアルタイム監視・バックアップする常駐アプリ
               → ログファイルを収集するだけ。DBには書かない。
StellaRecord   Polarisが収集したログを読み込み、SQLiteのDBを構築するアプリ
               → stellarecord.db を生成・管理する。手動起動。バックグラウンド動作ではない。

【Layer 2 / アプリ層】
Alpheratz      stellarecord.db を参照し、写真管理・検索を行うアプリ
               → 自前の alpheratz.db も持つ。読み取りがメイン。
Mira           日記アプリ。stellarecord.db を参照する予定。未着手。
```

Layer 2アプリは **stellarecord.db を参照して各自のDBを作る** という2階層構造になっている。
Layer 1とLayer 2の間には「DBスキーマの契約」があり、StellaRecordがスキーマを変えると全Layer 2が壊れるリスクがある。

## データ・ファイルの配置方針（重要）

インストール先（exeの場所）とデータの場所は**別物**である。混同しないこと。

```
【インストール先 — exeの場所】
C:\Program Files\CosmoArtsStore\
├── Polaris\polaris.exe
├── StellaRecord\stellarecord.exe
└── Alpheratz\alpheratz.exe

※ NSISインストーラーでユーザーが変更可能。データは一切ここに置かない。
```

```
【データの場所 — AppData\Local 固定】
C:\Users\{ユーザー名}\AppData\Local\CosmoArtsStore\
├── cosmos.json          ← エコシステム共通設定ファイル（StellaRecordが作成・管理）
├── polaris\
│   └── polaris.log
├── stellarecord\
│   └── stellarecord.db  ← デフォルト配置（ユーザーが変更可能）
└── alpheratz\
    └── alpheratz.db
```

`AppData\Local` はWindowsのユーザープロファイルに紐づいており、exeのインストール先がどこに変わっても常に同じパスになる。ここを使う理由はDBが大容量になりうるため（`Roaming` は同期を前提とした領域のため不適切）。

### cosmos.json の役割

Layer 2アプリはstellarecord.dbのパスを直接知る手段がない。StellaRecordが `cosmos.json` を書き、Layer 2アプリはこれを読むことでDBの場所を知る。

```json
{
  "stellarecord_db": "C:\\Users\\foo\\AppData\\Local\\CosmoArtsStore\\stellarecord\\stellarecord.db"
}
```

ユーザーがDBを別ドライブに移動した場合もここだけ書き換わる。Layer 2アプリはこのファイルが存在しない場合、「先にStellaRecordを起動してください」とユーザーに案内してクラッシュしないようにすること。

## ⚠️ 最優先 — 実行不能・衝突・データ破損リスク

### #1 アプリケーション識別子の衝突
✅ **修正する**

**問題：** 全アプリの `tauri.conf.json` の `identifier` が `com.cosmoartsstore.stellarecord` で統一されており衝突している。Windowsは `AppData` フォルダやレジストリをこのIDで管理するため、別アプリが同じIDを持つと設定・データが上書き・衝突する。

**修正内容：** 各アプリのIDを以下に変更する。

```
Polaris      → com.cosmoartsstore.polaris
StellaRecord → com.cosmoartsstore.stellarecord  （これが正とする）
Alpheratz    → com.cosmoartsstore.alpheratz
Mira         → com.cosmoartsstore.mira
```

**注意：** インストールフォルダ（Program Files配下）は `CosmoArtsStore` で統一したままでよい。identifierはデータ管理の話であり、インストール先の話ではない。

---

### #2 ハードコードされた古いパス / Fドライブ絶対パス
✅ **修正する**

**問題：** `package.json` のビルドスクリプト等に `F:\dev\RE-NAME-SYS\...` のような絶対パスが残っている。これは開発者本人のPC以外でビルドが通らない原因になる。また旧プロジェクト名 `RE-NAME-SYS` や `STELLARECORD`（大文字）という参照が `db.rs` / `config.rs` 等に残存している。

**修正内容：**
- Fドライブ等の絶対パスを相対パスまたは環境変数に置き換える
- `RE-NAME-SYS` / `STELLARECORD`（旧名）への参照を現在の正しいパスに修正する

---

### #3 未実装のコア機能（Alpheratz）
✅ **修正する**

**問題：** AlpheratzのフロントエンドUIには「CLIPによる画像検索」「DBSCANクラスタリング」のボタンが存在するが、Rust側の実装が一切ない。ボタンを押しても何も起きない。

**修正内容：** Rust側にCLIP検索・DBSCANクラスタリングのロジックを実装し、フロントエンドのボタンと接続する。

---

### #4 二重起動防止ロジックの欠落
✅ **修正する**

**問題：** 複数のインスタンスが同時起動すると同一のSQLite DBに同時書き込みが発生し、データが破損するリスクがある。

**修正対象：** DBへの書き込みが発生する **PolarisとStellaRecordの2アプリのみ** にMutexによる二重起動防止を追加する。

**注意：** AlpheratzとMiraは読み取りメインのため今回は対象外。Discordのような「起動していたら前のを使う」挙動と同じ仕組みでよい。

---

### #5 プロセス管理の矛盾（Planetarium残存）
✅ **修正する**

**問題：** かつて `Planetarium` は独立した `.exe` だったが、現在はStellaRecordのモジュールとして統合されている。にもかかわらずコード内に `planetarium.exe` を終了させようとする処理が残っており、存在しないプロセスを操作しようとしてエラーになる。

**修正内容：**
- コード内の `Planetarium` という名称・参照を完全に削除し、`StellaRecord` で統一する
- `planetarium.db` というファイル名が残っている場合は `stellarecord.db` に改名する
- 既存テスターがいる場合は再インストールを案内するか、古いDBファイルが存在すれば新しい名前に移行するマイグレーション処理を追加すること

### #20 環境変数取得の失敗ハンドリング（unwrap_or_default）
🔍 **要確認**

**問題：** `std::env::var("LOCALAPPDATA").unwrap_or_default()` のような記述があると、取得失敗時に空文字列が返り、`\stellarecord\db.sqlite` のような不正なパスでファイル操作を試みてクラッシュや予期しない場所へのファイル生成が起きる。

**確認内容：** `LOCALAPPDATA` / `APPDATA` 等の環境変数取得箇所を全て確認する。

**修正方針：** `unwrap_or_default()` を使っている箇所は、失敗時に早期エラーリターンする形に変更する。

```rust
// NG
let path = std::env::var("LOCALAPPDATA").unwrap_or_default();

// OK
let path = std::env::var("LOCALAPPDATA")
    .map_err(|_| AppError::EnvVarNotFound("LOCALAPPDATA"))?;
```

### #21 Windows APIクレートの混在（winapi / windows）
✅ **修正する**

**問題：** 古い `winapi` クレートと現代的なMicrosoft公式の `windows` クレートがプロジェクト間で混在している。`winapi` はメンテナンスがほぼ停止しており、両方が依存関係に入るとバイナリサイズも無駄に増える。

**修正内容：** `winapi` クレートへの依存を全て削除し、`windows` クレートに統一する。
