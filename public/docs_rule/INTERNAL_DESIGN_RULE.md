# 内部設計書（詳細設計書）作成ガイドライン：Agent用指示書

本ドキュメントは、今後システム内のモジュールや機能について「内部設計書」を作成する際の Agent（AIアシスタント）向けの動作規定である。
実装者が迷うことなく、かつ保守性の高いコードを生成するための「施工図」としての品質を担保することを目的とする。

## 1. 内部設計の定義と目的
- **定義**: 基本設計（何を作るか）を「どう実現するか」という実装レベルまで落とし込んだ技術文書。
- **目的**: 
    - 実装時の手戻り防止（ロジックの不備を設計段階で発見）。
    - チーム内（または将来の自分）への正確な実装指針の提供。
    - 保守・改修時のブラックボックス化防止。

## 2. 推奨される粒度（グラニュラリティ）
「プログラムの日本語訳」ではなく**「実装の方針と処理の流れ」**を記述する。

- **書くべきこと（高粒度）**:
    - **具体的なデータ形式**: 正規表現パターン、JSONスキーマ、DBカラム型。
    - **主要なアルゴリズム**: 特定のライブラリ関数、条件分岐の判定基準。
    - **例外処理（重要）**: バリデーションエラー、通信失敗、タイムアウト時の挙動。
    - **入出力インターフェース**: 関数への引数、返り値、副作用（DBへの書き込み等）。
    - **状態遷移**: FrontendのState管理、Backendのステータス管理。
- **書かなくてよいこと（低粒度）**:
    - 変数の型宣言や単純なループの構文。
    - フレームワーク標準の挙動（標準的なCRUD等）。
    - 言語固有のボイラープレート。

## 3. 必須構成要素と言及レベル
設計書作成時には以下の項目を網羅すること（[detailed design document sample.md](file:///f:/DEVELOPFOLDER/RE-NAME-SYS/public/detailed%20design%20document%20sample.md) を標準とする）。

### 3.1 概要・背景
- 処理の目的、ビジネス的な要求（なぜこの機能が必要か）を簡潔に記述。

### 3.2 データ・データベース設計
- 使用するテーブル、インデックス、初期化PRAGMA（WALモード等）を明記。
- データの制約（PRIMARY KEY, UNIQUE, DEFAULT値）への言及。

### 3.3 処理フロー（フェーズ分け）
- 処理全体を明確なフェーズ（Phase 1: 収集, Phase 2: 照合, Phase 3: 登録等）に分割。
- 非同期処理（async/await）の有無と、その起動タイミング（マウント時、ボタン押下時等）。

### 3.4 内部ロジックの詳細
- **Input/Process/Output/Exception** の4要素を意識する。
- 曖昧な表現（「エラーを表示する」等）を避け、具体的な方法（「toast表示」「赤文字でフィールド下」等）を指定。

### 3.5 フロントエンド・UIロジック
- Reactの状態変数（useState）とライフサイクル（useEffect）の役割分担。
- パフォーマンス対策（Virtual Scrollの計算ロジック、メモ化等）の記述。

### 3.6 シーケンス・ダイアグラム
- コンポーネント間、またはFrontend-Backend間の通信順序を Mermaid（mermaid-js）で視覚化。

## 4. 記述時の品質チェックリスト
- [ ] **具体性**: 悪い例「データを保存する」→ 良い例「JSONに変換し、SQLiteのphotosテーブルにINSERT OR IGNOREで登録する」。
- [ ] **網羅性**: 正常系だけでなく、データが空の場合や接続エラー時の処理が書かれているか。
- [ ] **視覚性**: 表形式（Table）や図解（Mermaid）を活用し、一目で全体像が掴めるか。
- [ ] **整合性**: 仕様書や基本設計とのズレがないか。

## 5. 運用ルール
- 内部設計書は常に最新の実装状態を反映させる。
- 実装中に設計の不整合が見つかった場合は、先に設計書を修正してからコードを修正するフロー（Design-First）を推奨する。
