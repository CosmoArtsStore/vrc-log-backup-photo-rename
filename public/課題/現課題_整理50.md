## 1. バックエンド (Rust/Tauri) - 30項目

#### 発見した問題 #1: DBスキーマの二重管理
**ファイル:** `Alpheratz/src-tauri/src/db.rs`
**行番号:** 65-101
**該当コード:**
```rust
     conn.execute_batch(
         "PRAGMA journal_mode = WAL;
          ...
          CREATE TABLE IF NOT EXISTS photos ( ... );"
     )
```
**問題の内容:** `migrations/V1__initial_schema.sql` が存在するにもかかわらず、`db.rs` 内でハードコードされた SQL でスキーマを定義している。
**影響範囲:** スキーマ変更時に2箇所の修正が必要になり、デッドコードや不整合の原因となる。
**推奨対応:** マイグレーションライブラリ（Refinery等）を正しく構成し、インラインSQLでの定義を削除する。

#### 発見した問題 #2: エラーメッセージ文字列への依存
**ファイル:** `Alpheratz/src-tauri/src/db.rs`
**行番号:** 53
**該当コード:**
```rust
if message.contains("duplicate column name")
```
**問題の内容:** `rusqlite` のエラーメッセージ文字列に直接依存して重複カラム判定を行っている。
**影響範囲:** ライブラリのアップデートや言語設定等によりメッセージが変わると、正常に動作しなくなる可能性がある。
**推奨対応:** `PRAGMA table_info` を使用して事前にカラムの存在を確認するか、エラーコード（ErrorCode）でマッチングする。

#### 発見した問題 #3: get_alpheratz_install_dir の重複定義
**ファイル:** `Alpheratz/src-tauri/src/utils.rs` (他2箇所)
**行番号:** 36
**該当コード:**
```rust
pub fn get_alpheratz_install_dir() -> Option<PathBuf> { ... }
```
**問題の内容:** `config.rs`, `db.rs`, `utils.rs` の3箇所に同一または類似のパス取得ロジックが重複して存在する。
**影響範囲:** インストールパス解決ロジックが分散し、エラーハンドリング（fail-fast/fail-open）の方針が一貫しなくなる。
**推奨対応:** `utils.rs` または `config.rs` の一方に集約し、他からはそれを呼び出す。

#### 発見した問題 #4: async内での同期的な画像処理 (Compute phash)
**ファイル:** `Alpheratz/src-tauri/src/scanner.rs`
**行番号:** 291
**該当コード:**
```rust
let computed = tauri::async_runtime::spawn_blocking(move || compute_phash(&path))
```
**注意:** これは `spawn_blocking` を使用するように修正済み。しかし、`analyze_photo` 内の `read_image_features`（L212）は未だ同期実行されている可能性がある。
**問題の内容:** `do_scan` ループ内で `analyze_photo` -> `read_image_features` を呼び出しており、大量ファイルの分析時にUIスレッドをブロックする。
**推奨対応:** `analyze_photo` 内の重い計算処理も `spawn_blocking` の対象とする。

#### 発見した問題 #5: DBインデックスの不足
**ファイル:** `Alpheratz/src-tauri/src/db.rs`
**行番号:** 98-99
**問題の内容:** `timestamp` と `world_name` にはインデックスがあるが、`phash` 検索や `is_favorite` フィルタリングを多用する場合、これらへのインデックスが不足している。
**推奨対応:** ユースケースに応じて `phash` や `is_favorite` へのインデックス追加を検討。

#### 発見した問題 #6: ログの無限肥大
**ファイル:** `Alpheratz/src-tauri/src/utils.rs`
**行番号:** 42
**該当コード:**
```rust
match OpenOptions::new().create(true).append(true).open(&path)
```
**問題の内容:** `info.log` に対して常にアペンドしており、ローテーション機能がない。
**影響範囲:** 長期運用時にログファイルが数GBに達し、ディスク容量を圧迫したり、ログの読み込みが困難になる。
**推奨対応:** `tracing-appender` 等を使用したサイズ/日付ベースのローテーション導入。

#### 発見した問題 #7: パス区切り文字の手動置換

### 調査結果サマリー

- **発見した問題の総数:** 50
- **重点課題:**
    - DB構成の二重化による保守性の低下
    - 同期的な計算処理による UI ブロック
    - フロントエンドでの型安全性の欠化
    - セキュリティ設定 (CSP) の放置
- **総評:** 基本的な機能は動作しているが、長期間の運用や大量のデータ（写真）を扱う際に表面化する「作りの甘さ」が 50 箇所特定された。
