## 総評

全体的にかなり丁寧に書かれていて、Tauri 2.x のベストプラクティスも押さえられています。ただ、**重複コード**と**DB管理の二重定義**がいくつかあって、今後のメンテで地雷になりそうです。

---

## 👍 良い点

**`LazyLock<Regex>`の使用**　静的Regexを正しく初期化できてます。以前の会話で決めた方針がちゃんと反映されてますね。

**バッチinsert + トランザクション**　`PHOTO_INSERT_BATCH_SIZE = 50` でまとめてコミット、pHashも同様。大量ファイル時のパフォーマンスを意識した設計です。

**キャンセル対応**　`AtomicBool` でスキャン中断を実装、キャンセルチェックのタイミングも適切です。

**WAL + `synchronous = NORMAL`**　同時アクセスへの準備もできてます。

**パニックフック + WindowsメッセージボックスでUX対応**　リリースビルドでのサイレントクラッシュ防止。これは地味にえらい。

**シンボリックリンクスキップ**　`symlink_metadata` でチェックしているのも安全側に倒した判断です。

## ⚠️ 問題点（優先度順）

### 🔴 高優先

**`warn_row_error` が `db.rs` と `scanner.rs` に重複定義**
まったく同じ関数が2箇所に存在します。`utils.rs` に移して `pub` にするべきです。

**`get_alpheratz_install_dir` が3箇所に定義**
`config.rs`、`db.rs`、`utils.rs` それぞれに独自実装があります。`utils.rs` の1箇所に集約して他から呼ぶ構造にすべきです。

**DBスキーマが二重定義**
`migrations/V1__initial_schema.sql` が存在するのに、`db.rs::init_alpheratz_db()` の中でも `execute_batch` でスキーマを再定義しています。マイグレーションファイルが**デッドコード**になっています。Refineryを使う方針だったはずが、インラインSQLに逆戻りしている状態です。

**`add_column_if_missing` のエラー文字列マッチング**
```rust
if message.contains("duplicate column name")
```
rusqliteの内部エラーメッセージ文字列に依存していて脆弱です。`rusqlite::Error` の型でマッチするか、`PRAGMA table_info()` で事前チェックする方が堅牢です。

---

### 🟡 中優先

**`get_photos` のCOUNT取得が意味をなしていない**
```rust
let row_count: usize = conn.query_row("SELECT COUNT(*) FROM photos", [], ...)
```
フィルターなしで全件カウントしてから、フィルター付きクエリを実行しています。`Vec::with_capacity` のヒントがフィルター後の実件数と一致しません。`Vec::new()` で十分です。

**`scan:completed` イベントが2回emit される**
`do_scan` の末尾と、`compute_missing_phashes_bg` の末尾それぞれで `scan:completed` を emit しています。フロントエンドで二重に完了処理が走る可能性があります。

**`world_exact = "unknown"` のハードコード文字列比較**
```rust
if world_exact.as_ref().map(|s| s.as_str()) == Some("unknown")
```
フロントエンドとバックエンドで文字列を合わせ続ける必要があり壊れやすいです。`enum WorldFilter { Exact(String), Unknown }` のような型で表現するのが理想です。

**`histogram` フィールドが実装されていない**
DBスキーマに `histogram BLOB` がありますが、計算・保存する処理がコード内に存在しません。今後実装予定のものと理解していますが、`is_favorite` も同様（DBにはあるがモデルに未反映）です。

---

### 🟢 低優先

**`tokio` がCargo.tomlにあるが直接未使用**
`tauri::async_runtime::spawn_blocking` を使っているので `tokio` は不要です。ビルド時間が増えるだけなので削除できます。

**ログローテーションなし**
`info.log` が無限に肥大します。写真管理アプリで長期運用するなら、サイズ上限や日付ローテーションを検討した方がいいですね。

**`backfill_missing_world_info` が毎回全件スキャン**
`world_id IS NULL` の写真を毎スキャン時に全件処理します。写真が数千枚規模になると体感できるほど遅くなる可能性があります。

## 現行コード照合メモ（2026-03-13）

上記レビューを、現在の `Alpheratz` コードと照合した補足を以下に残す。

### 採用したほうがよい指摘

#### 1. `useScan` の副作用ループ
- **該当ファイル:** `Alpheratz/src/hooks/useScan.ts`
- **判断:** 採用
- **理由:** `startScan` が `scanStatus` に依存し、その `startScan` を `useEffect` が依存配列に持っているため、完了やエラーで state が変わるたびに effect が再実行され、再スキャンを誘発しうる。これは実害のある問題。
- **対応方針:** `scanStatus` を多重起動防止の根拠にせず、`ref` ベースの実行中フラグへ切り替える。あわせて初期化 `useEffect` はマウント時 1 回だけ動かし、以後の更新は手動操作に限定する。
- **実施内容:** `startScan` から `scanStatus` 依存を外し、`isScanningRef` を導入した。`scan:completed` / `scan:error` 受信時に `isScanningRef.current = false` を戻す構造へ変更済み。これにより「初回自動開始は 1 回のみ」「完了後の再スキャンは起きない」「手動更新だけが再実行トリガー」という設計に揃えた。

#### 2. DB スキーマの二重管理
- **該当ファイル:** `Alpheratz/src-tauri/src/db.rs`, `Alpheratz/src-tauri/migrations/V1__initial_schema.sql`
- **判断:** 採用
- **理由:** 現在は `db.rs` 側で実際の初期化を行っており、SQL ファイルとの二重管理になっている。将来の変更漏れを生みやすい。

#### 3. `add_column_if_missing` の文字列一致判定
- **該当ファイル:** `Alpheratz/src-tauri/src/db.rs`
- **判断:** 採用
- **理由:** `"duplicate column name"` というエラーメッセージ文字列への依存は脆い。`PRAGMA table_info()` などで事前判定する方が堅牢。

#### 4. `COUNT(*)` の capacity 見積もり
- **該当ファイル:** `Alpheratz/src-tauri/src/db.rs`
- **判断:** 低優先で採用
- **理由:** フィルタなし全件数を `Vec::with_capacity` に使っており、最適化としては精度が低い。ただし重大な問題ではない。

### 反論できる指摘

#### 1. `warn_row_error` 重複を高優先とする評価
- **該当ファイル:** `Alpheratz/src-tauri/src/db.rs`, `Alpheratz/src-tauri/src/scanner.rs`
- **判断:** 反論可
- **理由:** 重複は事実だが、この関数は「共通業務ルール」ではなく、それぞれの層のローカルな補助関数として置かれている。`db.rs` では DB 行デコード失敗を局所的に `WARN` へ落とすため、`scanner.rs` ではスキャン系クエリや pHash 対象行の失敗を継続処理に変換するために使っており、呼ばれる文脈も責務も異なる。
- **補足:** この種の小さな局所ヘルパを無理に `utils.rs` へ集約すると、逆に「汎用 utility が増えすぎて責務がぼやける」副作用がある。特に Alpheratz は「層ごとに失敗の扱いを明確にする」方針なので、各層の近くに置くこと自体は設計として成立している。
- **結論:** 共通化は「やってもよい改善」だが、優先度を上げて潰すべき構造欠陥ではない。Claude に返すなら「DRY違反というより局所ヘルパの配置戦略の違い」であり、高優先扱いは過大評価と説明できる。

#### 2. `get_alpheratz_install_dir` 重複を高優先とする評価
- **該当ファイル:** `Alpheratz/src-tauri/src/config.rs`, `Alpheratz/src-tauri/src/db.rs`, `Alpheratz/src-tauri/src/utils.rs`
- **判断:** 反論可
- **理由:** 確かに重複はあるが、これは「設計整理の課題」であって「直ちにバグを生む欠陥」ではない。現時点の実害は `useScan` の副作用ループや DB スキーマ二重管理の方が明確であり、優先度はそちらが上。
- **補足:** さらに、`config.rs` / `db.rs` / `utils.rs` はそれぞれ fail-open / fail-fast / logging の方針が微妙に異なるため、単純に 1 箇所へ寄せると「どの層の失敗方針を正とするか」を再設計する必要がある。つまり、単なるコピペ削除では終わらず、むしろ慎重にやるべき整理である。
- **結論:** 将来的な集約候補ではあるが、今この段階で高優先に据えるのは適切ではない。Claude に渡す補足としては「重複そのものは認めるが、優先度評価には同意しない」が妥当。

### 保留扱いの指摘

#### 1. `world_exact = "unknown"` の型化
- **該当ファイル:** `Alpheratz/src-tauri/src/db.rs`, `Alpheratz/src/hooks/usePhotos.ts`
- **判断:** 保留
- **理由:** 理想的には enum の方が安全だが、現状の Tauri command 境界では sentinel 文字列運用でも十分に成立している。今すぐの採用対象でも、強い反論対象でもなく、後続の API 整理時にまとめて再検討するのが妥当。

### 補足

- 外部 AI の評価には妥当なものが含まれている一方で、コードの新しい状態を完全には反映していない部分もある。
- 特に `STELLA RECORD` 連携や外部 URL 起動は既に撤去済みであり、その前提での評価は現在には当てはまらない。
- また、`scan:completed` 二重 emit の指摘は表面的には近いが、本質的な問題はフロント側 `useScan` の副作用ループにある。

### 現時点の判断

- **今すぐ直す価値が高い:** `useScan` の再実行構造、DB スキーマ一本化、`add_column_if_missing` の堅牢化
- **後回しでよい:** `warn_row_error` 共通化、install_dir 解決共通化、`COUNT(*)` 最適化
- **保留でよい:** `world_exact = "unknown"` の型化
