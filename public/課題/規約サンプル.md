## 4.1 命名規則（Rust）

Rust Style Guide（公式）をベースラインとし、rustfmt 標準に整合させる。

| 識別子の種類 | スタイル | 本プロジェクトの例 |
|---|---|---|
| struct / enum / trait / type alias | UpperCamelCase | `PolarisSetting`, `PlanetariumSetting`, `AppError` |
| fn（関数・メソッド） / 変数 / モジュール | snake_case | `load_setting`, `backup_logs`, `parse_and_import` |
| const / static | SCREAMING_SNAKE_CASE | `SCHEMA`（DB定義定数）, `RE_TIME`（正規表現） |
| ファイル名 | snake_case.rs | `config.rs`, `logger.rs`, `main.rs` |

**【規約】命名統一**  
`Lazy<Regex>` 変数は `RE_` プレフィックスで統一する（現実装に即した規約）。モジュール名は責務を表す単語を snake_case で命名し、略称は避ける。

## 4.2 コードスタイル（rustfmt / Clippy）

### 4.2.1 rustfmt

`cargo fmt` を実行して整形を強制する。設定は最小限とし、スタイル議論のコストを削減する。

```bash
# フォーマット確認（CI 用）
cargo fmt -- --check

# 自動整形（開発時）
cargo fmt
```

| 設定項目 | 値 | 備考 |
|---|---|---|
| 最大行幅 (max_width) | 100 | Rust Style Guide デフォルト |
| インデント | スペース 4 | Rust Style Guide デフォルト |
| rustfmt.toml | 原則作成しない | 既定スタイルで十分な場合はオーバーライドしない |

### 4.2.2 Clippy

Clippy は「コード品質ゲート」として CI に組み込む。警告は原則として修正し、例外的に無視する場合は `#[allow(clippy::...)]` にコメントで理由を記す。

```bash
# CI での実行例（警告をエラーとして扱う）
cargo clippy -- -D warnings
```

## 4.3 エラーハンドリング規約

### 4.3.1 Display メッセージ形式

Rust API Guidelines に従い、`Display` で表示するエラーメッセージは以下の形式に統一する。

| ルール | OK 例 | NG 例 |
|---|---|---|
| 小文字で始める (lowercase) | `"failed to open archive"` | `"Failed to open archive"` |
| 末尾に句読点を付けない | `"setting file not found"` | `"設定ファイルが見つかりません。"` |
| 簡潔に (concise) | `"invalid log format"` | `"ログのフォーマットが正しくありません"` |

### 4.3.2 Result 型の使用方針

`panic!`（`.unwrap()` / `.expect()`）はシステム不変条件が壊れた場合のみ許容する。通常のエラーパスは `?` 演算子で `Result` を伝搬する。

| 箇所 | 現状 | 方針 |
|---|---|---|
| DB 接続失敗 (Planetarium) | `?` で main() まで伝搬 → exit | 現状維持（適切） |
| 設定ファイル読み込み失敗 | Default 値で続行 | 現状維持（ログ出力を追加推奨） |
| タスクトレイビルド失敗 (Polaris) | `.expect()` でパニック | 現状維持（システム不変条件） |
| 個別ファイルコピー失敗 (Polaris) | count 増やさずスキップ（ログなし） | 警告ログ出力を追加推奨 |
| tar.zst 失敗 (Planetarium) | eprintln して続行 | 構造化ログへの移行を推奨 |

## 4.4 SQLite 設計規約

### 4.4.1 接続初期化 PRAGMA

接続確立直後に以下の PRAGMA を実行することを原則とする。

| PRAGMA | 推奨値 | Planetarium 現状 | 方針 |
|---|---|---|---|
| journal_mode | WAL | 設定済み ✅ | 現状維持 |
| foreign_keys | ON | 未設定 ⚠（SCHEMA に FK 定義あり） | 次バージョンで有効化を検討 |
| synchronous | NORMAL（WAL時） | 未設定 | WAL環境では NORMAL が適切 |

> ⚠ Planetarium.exe は SCHEMA に FOREIGN KEY 定義を含むが、現在 `PRAGMA foreign_keys = ON` が発行されていない。FK 制約が機能しないため、参照整合性はアプリケーション層で担保している状態。次バージョンで追加することを推奨する。

```rust
// 推奨: 接続直後に一括設定
conn.execute_batch("
    PRAGMA journal_mode = WAL;
    PRAGMA foreign_keys = ON;
    PRAGMA synchronous = NORMAL;
")?;
```

### 4.4.2 SQL インジェクション対策

ユーザー入力・外部データを SQL に渡す場合は必ずバインド変数（`?` または `?N`）を使用し、文字列連結を禁止する。

| 分類 | 規約 | 現状 |
|---|---|---|
| INSERT / UPDATE / SELECT | `?` プレースホルダでバインド | 実装済み ✅ |
| 動的テーブル名・カラム名 | 使用禁止（スキーマは定数 SCHEMA で管理） | 実装済み ✅ |
| ユーザー入力の文字列連結 | 禁止 | 実装なし ✅ |

### 4.4.3 トランザクション方針

1 ファイル分のパース・DB 登録処理は単一トランザクション内で完結させる。失敗時は自動ロールバックにより DB 整合性を維持する。

> 📌 `parse_and_import()` は `conn.transaction()` → `tx.commit()` / 自動ロールバックの構造で実装されており、この方針を継続すること。複数ファイルを跨ぐトランザクションは避ける（部分失敗の影響範囲を限定するため）。

## 4.5 ログ規約

### 4.5.1 ログフォーマット（Polaris）

`polaris_appinfo.log` は以下のフォーマットで統一されている（logger モジュール）。

```
[YYYY-MM-DD HH:MM:SS] [モジュール名] メッセージ

例: [2024-01-15 22:30:45] [Polaris] VRChat 起動検知
```

| 規約項目 | 内容 |
|---|---|
| モジュール名フィールド | 呼び出し元を識別するために必須。現在は `"Polaris"` 固定 |
| ログレベル表現 | 通常は省略。警告は `[WARNING]` プレフィックスをメッセージ先頭に付与 |
| PII / 秘密情報 | ログに出力禁止（パスワード・トークン・個人識別可能な情報） |
| ファイルオープン競合対策 | `FILE_SHARE_READ \| WRITE \| DELETE` フラグで他プロセスとの共存を確保（実装済み） |

### 4.5.2 Planetarium の標準出力ログ

Planetarium.exe は Tauri を使用しないため、進捗・完了通知を stdout への `println!` で行う。STELLA_RECORD.exe がポーリングして進捗を把握する設計。

| 出力内容 | フォーマット | 用途 |
|---|---|---|
| 処理開始 | `[Planetarium] [PROGRESS] 0%` | 進捗通知 |
| パース中 | `[Planetarium] [STATUS] パース中... N 行` | 行数報告（1000行ごと） |
| 処理完了 | `[Planetarium] [PROGRESS] 100%` | 完了通知 |
| 最終完了 | `[Planetarium] 処理完了` | UI側のポーリング終了検知 |
| 強制Sync完了 | `強制Sync完了` | 強制Sync終了の検知 |

> ⚠ 完了通知はプレーンテキストであり JSON 形式ではない。フォーマット変更は STELLA_RECORD.exe 側のパーサーに影響するため、変更時は両アプリを同時にリリースすること。

## 4.6 テスト戦略

`cargo test` を品質ゲートとして CI に組み込む。

| テスト種別 | 配置場所 | 対象 | コマンド |
|---|---|---|---|
| ユニットテスト | `src/` 内の `#[cfg(test)]` モジュール | 個別関数・ロジック | `cargo test` |
| 統合テスト | `tests/` ディレクトリ | モジュール間連携・DB 操作 | `cargo test` |
| ドキュメントテスト | `///` コメント内の ` ``` ` ブロック | 公開 API の使用例 | `cargo test --doc` |

### 4.6.1 優先してテストすべき領域

| 領域 | 理由 | テスト対象例 |
|---|---|---|
| パース正規表現（RE_*） | VRChat ログ仕様変更の影響を早期検知 | RE_JOINING、RE_PLAYER_JOIN 等の各パターン |
| マイグレーション処理 | データ損失リスクが高い | run_migration の ZIP/ディレクトリ各パス |
| Privacy/Tracking モードのマスク | プライバシー保護の核心 | プレイヤー名マスク、my_display_name 保存 |
| 設定ファイルのデシリアライズ | フィールド追加時の後方互換 | デフォルト値・serde(default) の動作 |
| DB スキーマ作成 | execute_batch(SCHEMA) の安全性 | 全テーブルの CREATE IF NOT EXISTS |

### 4.6.2 カバレッジ計測

`cargo-llvm-cov` を使用してカバレッジを計測する。100% を機械的に強制するより、上記の優先領域を重点的にカバーする方針とする。

```bash
# cargo-llvm-cov のインストール
cargo install cargo-llvm-cov

# カバレッジレポート生成
cargo llvm-cov --html
```

## 4.8 セキュリティ規約

### 4.8.1 パス解決

すべてのパスは環境変数（`%LOCALAPPDATA%` / `%APPDATA%`）を起点に解決し、ハードコードされた絶対パスを禁止する（両アプリとも実装済み）。

| 規約 | 根拠 | 現状 |
|---|---|---|
| `%LOCALAPPDATA%` を `std::env::var()` で取得 | 異なるユーザープロファイル環境での動作保証 | 実装済み ✅ |
| install_dir 相対パスを使用しない | UAC・インストールディレクトリ変更への耐性 | 実装済み ✅ (§1.2 / §2.2) |
| パス取得失敗時は Err を返し、unwrap() しない | クラッシュよりも安全なフォールバック | 一部実装済み（要確認） |

### 4.8.2 ファイル共有モード（Windows）

Windows API の ShareMode は「必要最小限のフラグ」のみ設定する。

| 対象 | 設定フラグ | 理由 |
|---|---|---|
| polaris_appinfo.log の書き込み (Polaris) | `FILE_SHARE_READ \| WRITE \| DELETE` | STELLA_RECORD.exe が同時読み取り可能にするため |
| output_log_*.txt の読み取り (Planetarium) | `FILE_SHARE_READ` のみ | VRChat が書き込み中でも読み取り可能にするため |

> ⚠ `FILE_SHARE_WRITE` を設定すると、他プロセスが同時にファイルを書き換えられる。`polaris_appinfo.log` は Polaris.exe のみが書き込むことを前提とし、STELLA_RECORD.exe による書き込みを行わない設計を維持すること。

### 4.8.3 機密情報・PII の取り扱い

| 分類 | 規約 |
|---|---|
| ユーザーの usr_ID / 表示名 | Privacy モード時は DB 保存・ログ出力ともにマスク処理を適用 |
| ログ出力への PII 含有禁止 | polaris_appinfo.log / stdout いずれにも個人情報を含めない |
| 設定ファイルのパスワード・トークン | PolarisSetting.json / PlanetariumSetting.json には認証情報フィールドを追加しない |

## 4.9 CI/CD パイプライン方針

GitHub Actions を使用し、以下のステージ構成を推奨する。`GITHUB_TOKEN` は最小権限（リポジトリへの読み取り）で運用し、リリース時のみ write 権限を付与する。

| ステージ | チェック内容 | ツール |
|---|---|---|
| Static Checks | フォーマット確認 | `cargo fmt -- --check` |
| Static Checks | Lint（警告ゼロ必須） | `cargo clippy -- -D warnings` |
| Static Checks | 型チェック | `cargo check` |
| Security | 脆弱性チェック | `cargo audit` |
| Security | ライセンス・依存チェック | `cargo deny check` |
| Tests | ユニット / 統合 / doctest | `cargo test` |
| Build | Windows バイナリビルド | `cargo build --release` |

**【規約】品質ゲート**  
上記ステージのいずれかが失敗した場合、PR のマージを禁止する。フォーマット違反は CI で検知し、手元で `cargo fmt` を実行してから再プッシュする。Clippy 警告は修正必須とし、`#[allow(...)]` は理由コメントとセットで使用する。

## 4.10 コードレビュー基準

コードレビューの目的は「コードベースの健全性を時間とともに改善すること」とする。フォーマット・Lint は CI で自動検知するため、レビューは以下の観点に集中する。

| レビュー観点 | 確認項目 | 重要度 |
|---|---|---|
| 設計・責務 | モジュールの責務が明確か。単一責任原則を守っているか | 高 |
| エラー処理 | 失敗パスが網羅されているか。unwrap() の使用が適切か | 高 |
| セキュリティ | パス解決・ShareMode・PII の取り扱いが規約に従っているか | 高 |
| DB 操作 | バインド変数使用・トランザクション範囲が適切か | 高 |
| テスト | 新機能・バグ修正にテストが付いているか | 中 |
| ログ | PII を含まないか。ログレベルが適切か | 中 |
| パフォーマンス | 不要なファイル走査・メモリアロケーションがないか | 低 |

**【規約】Small PR 推奨**  
PR は原則として機能単位で分割し、1 PR = 1 責務の変更とする。ロールバック容易性を確保し、レビュー品質を上げるためにファイル数が多い PR は分割を検討する。
