# NSIS監査メモ

更新日: 2026-03-14
対象: `Polaris` / `StellaRecord` / `Alpheratz`

## 先に結論

現状の NSIS は 3 アプリともほぼ同じ Tauri 既定テンプレートを使っており、不審な常駐処理や隠し通信のようなものは見当たりませんでした。

ただし、1 点だけ実害が出る可能性が高い差異があります。

- `StellaRecord` の `hooks.nsi` は `taskkill /IM StellaRecord.exe` を呼んでいるが、実際の Rust パッケージ名と生成バイナリ名は `stella_record_ui.exe` です。
- そのため、インストール前とアンインストール前に既存プロセスを止めたい意図があっても、実際には停止できていない可能性があります。
- これにより、更新時に exe 上書きや削除が失敗する、または再起動待ちになる可能性があります。

## 監査した観点

- NSIS テンプレートで何をしているか
- 各アプリ独自の `hooks.nsi` が何を追加しているか
- 実行時コードがどのレジストリを参照しているか
- インストーラ設定と実行時挙動が一致しているか
- 不要なフォルダや不要な永続設定を埋め込んでいないか

## 共通テンプレートの動作

3 アプリの `windows/installer.nsi` は、細部の置換値を除けば同じ構造です。主な処理は以下です。

1. インストーラの言語、圧縮、DPI 設定を初期化する。
2. 既存インストールや旧 WiX 版があれば検出し、更新またはアンインストールの導線を出す。
3. WebView2 が必要なら導入確認を行う。
4. 本体ファイル、リソース、外部バイナリを `$INSTDIR` に配置する。
5. アンインストーラを生成し、Windows の「アプリと機能」に出すためのレジストリを書き込む。
6. スタートメニューとデスクトップショートカットを作る。
7. 必要に応じてインストール完了後にアプリを起動する。
8. アンインストール時は exe、リソース、ショートカット、アンインストール情報を削除する。

## インストール先の扱い

現在の既定インストール先は、全アプリで次に統一されています。

- ユーザー単位: `%LOCALAPPDATA%\\CosmoArtsStore\\STELLAProject\\{AppName}`
- 管理者インストール時: `%ProgramFiles%\\CosmoArtsStore\\STELLAProject\\{AppName}`

この方針自体は妥当です。`Polaris` 基準で揃っており、3 アプリが同じルート配下に入りつつ、アプリごとにフォルダは分離されています。

### 対応方針: インストール先の固定

- 3 アプリとも NSIS の install mode を `currentUser` 固定にする。
- 3 アプリとも既定インストール先を `%LOCALAPPDATA%\\CosmoArtsStore\\STELLAProject\\{AppName}` のみに固定する。
- `Program Files` 配下への配置は許可しない。理由は、現実装が設定、DB、ログ、キャッシュ、バックアップをインストール先直下で扱うためです。
- インストーラのディレクトリ選択でも、少なくとも `Program Files` 配下と書き込み制限が強い場所は避ける方針を明記する。
- セキュリティソフト誤検知を避ける観点でも、書き換えが頻発する実行ファイル直下を管理者領域へ置かない方針を採る。

## レジストリで何を書いているか

### 1. Windows 標準のアンインストール情報

各アプリは以下の系統のキーを書きます。

- `Software\\Microsoft\\Windows\\CurrentVersion\\Uninstall\\{ProductName}`

ここには主に以下が入ります。

- 表示名
- 表示アイコン
- バージョン
- 発行元
- インストール先
- アンインストールコマンド
- 推定サイズ
- 修復不可、変更不可フラグ

これは Windows の「アプリと機能」に表示するための標準情報で、不審ではありません。

### 2. CosmoArtsStore 系の補助キー

各アプリは以下の系統も使います。

- `Software\\CosmoArtsStore\\{ProductName}`

このキーは Tauri テンプレート側で、言語設定や前回インストール先の復元に使われます。これも標準的な補助用途です。

### 3. STELLAProject 独自の InstallLocation キー

各アプリは追加で以下を書きます。

- `HKCU\\Software\\CosmoArtsStore\\STELLAProject\\Polaris`
- `HKCU\\Software\\CosmoArtsStore\\STELLAProject\\StellaRecord`
- `HKCU\\Software\\CosmoArtsStore\\STELLAProject\\Alpheratz`

値名はすべて `InstallLocation` です。

この独自キーは実行時コードが直接参照しています。つまり、このキーは余分ではなく、今のアプリ設計では必須です。

## 実行時コードとの整合

### Polaris

`Polaris` は自分の `InstallLocation` を読み、以下をその配下に作ります。

- `archive`
- `info.log`

これはインストーラが書く `HKCU\\Software\\CosmoArtsStore\\STELLAProject\\Polaris\\InstallLocation` と一致しています。

### StellaRecord

`StellaRecord` は次の 2 つを参照します。

- 自分自身の `InstallLocation`
- `Polaris` の `InstallLocation`

用途は以下です。

- 自分の `info.log`、設定ファイル、DB を自分のインストール先配下に置く
- `Polaris.exe` の場所を `Polaris` のインストール先から解決する

この意味で、`Polaris` と `StellaRecord` のインストーラ設定は連携しています。

### Alpheratz

`Alpheratz` は自分の `InstallLocation` を読み、以下をその配下に置きます。

- `alpheratz.json`
- `cache`
- `info.log`
- `alpheratz.db`

加えて、`StellaRecord` のインストール先を読んで `stellarecord.db` に到達します。

`db.rs` では `STELLA_RECORD` と `StellaRecord` の両方を候補にしていますが、これは旧名互換のためのフォールバックと見てよく、不審挙動ではありません。

## 各アプリの独自 hooks

### Polaris

- インストール前: `Polaris.exe` を `taskkill` で停止
- インストール後: `HKCU\\Software\\Microsoft\\Windows\\CurrentVersion\\Run\\Polaris` を追加
- アンインストール前: `Polaris.exe` を停止

つまり Polaris だけがログイン時自動起動されます。常駐アプリとしては整合しています。

### StellaRecord

- インストール前: `StellaRecord.exe` を停止しようとする
- インストール後: 自動起動登録なし
- アンインストール前: `StellaRecord.exe` を停止しようとする

意図は正常ですが、停止対象名が実バイナリ名と一致していません。

### Alpheratz

- インストール前: `Alpheratz.exe` を停止
- インストール後: 自動起動登録なし
- アンインストール前: `Alpheratz.exe` を停止

Windows のプロセス名比較は通常大文字小文字を区別しないため、`alpheratz.exe` 実体に対しては問題になりにくいです。

### 対応方針: 自動起動の責務分離

- インストーラーで Run キーを直接登録するのは `Polaris` のみとする。
- `StellaRecord` と `Alpheratz` はインストーラーで自動起動登録しない。
- `StellaRecord` と `Alpheratz` にはアプリ設定画面から ON/OFF できる「ログイン時に起動」設定を追加する。
- 初回起動時は Discord のように、利用者が明示的に選べる UI を出す方針で整理する。
- 実装はアプリ側で `HKCU\\Software\\Microsoft\\Windows\\CurrentVersion\\Run` を管理し、インストーラーはその後始末だけを持つ。
## アンインストール時に消えるもの

共通テンプレートが削除する対象は概ね以下です。

- メイン exe
- インストール時に配置したリソース
- 外部バイナリ
- アンインストーラ自身
- スタートメニューショートカット
- デスクトップショートカット
- Windows のアンインストール情報キー
- STELLAProject 独自の `InstallLocation` キー

加えて、更新時以外は `HKCU\\Software\\Microsoft\\Windows\\CurrentVersion\\Run\\{ProductName}` も削除します。

この Run キー削除は 3 アプリ共通で入っていますが、実際に書いているのは `Polaris` だけです。`StellaRecord` と `Alpheratz` では空振りになるだけなので不審ではありませんが、やや汎用的すぎる処理です。


## 「Delete app data」チェックの実態

Tauri の標準テンプレートは、追加チェックが付くと以下を消します。

- `%APPDATA%\\{bundleId}`
- `%LOCALAPPDATA%\\{bundleId}`

ただし、このプロジェクトの主要データはその場所に寄っていません。

- `Polaris`: インストール先配下の `archive` と `info.log`
- `StellaRecord`: インストール先配下の `info.log`、設定 JSON、DB
- `Alpheratz`: インストール先配下の `alpheratz.json`、`cache`、`info.log`、DB

つまり、このチェックは「アプリ標準の AppData」を消すためのもので、STELLAProject 実装で使っている主要データ整理とはあまり連動していません。

今回の実装では、主要データはインストールフォルダ配下にあるため、インストール先を丸ごと削除する通常アンインストールの方が実質的には強く効きます。

## 不審挙動の有無

今回見た範囲では、以下のような不審挙動は確認していません。

- 見えない場所への恒久的な追加フォルダ作成
- 複数の謎のレジストリ散布
- 目的不明のサービス登録
- スケジュールタスク登録
- 不要なファイル関連付け
- 目的不明のネットワーク設定

残っているレジストリ操作は、主に以下の 3 つに集約されます。

- Windows のアンインストール情報
- インストーラの補助情報
- STELLAProject 各アプリのインストール先共有

## 要注意点

### 1. StellaRecord の停止対象名が実ファイル名と一致していない

最優先で修正候補です。

- 現在の hooks は `StellaRecord.exe`
- Rust パッケージ名は `stella_record_ui`
- NSIS 本体は `${MAINBINARYNAME}.exe` ベースで動いている

この差異だけは、更新・削除時の期待動作を壊す可能性があります。

### 対応方針: StellaRecord の構成整理

- `StellaRecord` は `F:\\DEVELOPFOLDER\\STELLAProject\\StellaRecord\\stella_record_ui` 配下に二重でぶら下がっており、他アプリと構成が揃っていません。
- `Alpheratz` と同様に、`StellaRecord` 直下へ `src-tauri`、フロントエンド `src`、`public`、`package.json` などを移し、単一アプリとして読める構成へ揃える。
- 構成整理後に、NSIS hooks の停止対象名を実バイナリ名に合わせて修正する。
- 可能なら hooks 側で固定文字列を減らし、`${MAINBINARYNAME}.exe` 基準で停止させる形へ寄せる。

### 2. アプリデータの保存先がインストールフォルダ直下

これは不審ではありませんが、運用上の注意点です。

- ユーザー単位インストールなら書き込み可能
- 管理者インストールで `Program Files` 配下に入った場合、通常権限での書き込みに失敗する可能性がある

特に `Polaris` の `archive`、`StellaRecord` の DB と設定、`Alpheratz` の DB と cache はこの影響を受けます。

将来「管理者インストールも正式対応する」なら、データ保存先は `LocalAppData` 側へ寄せた方が安全です。

### 3. Run キー削除は 3 アプリ共通だが、登録は Polaris のみ

これは害はありません。ただし、仕様書やコメント上では「Polaris のみ Run 登録する」と明記しておく方が理解しやすいです。

### 対応方針: Run キーの最終整理

- NSIS hooks で Run キーを `WriteRegStr` するのは `Polaris` のみとする。
- `StellaRecord` と `Alpheratz` は Run キーを自分で追加しない。
- ただしアンインストーラ側の `DeleteRegValue HKCU "Software\\Microsoft\\Windows\\CurrentVersion\\Run" "{ProductName}"` は残してよい。
- 理由は、将来アプリ側設定で Run キーを作った場合でも、アンインストール時に掃除できるためです。

## 整合性の評価

### 揃っている点

- 既定インストール先の方針
- 独自 `InstallLocation` キーの命名
- Polaris の自動起動方針
- アプリアイコンと NSIS アイコン指定
- ウィザード画像未使用化

### まだ揃えた方がよい点

- `StellaRecord` の停止対象 exe 名
- 管理者インストール時のデータ保存先戦略
- 「Delete app data」チェックが本当に何を消すかの説明

## 推奨アクション

1. `StellaRecord` の `hooks.nsi` を実バイナリ名基準へ修正する。
2. 3 アプリ共通で「インストール先に置くもの」と「将来 AppData に逃がすべきもの」を整理する。
3. `Polaris` だけ Run キーを使う理由を、インストーラコメントか公開ドキュメントに残す。
4. ユーザー向け説明では、「アンインストールで消えるもの」と「運用上残したいデータ」を明示する。

## 対応順序

1. `StellaRecord` のディレクトリ構成を `Alpheratz` と同じ粒度に揃える。
2. 3 アプリの NSIS を `currentUser` 固定、`LocalAppData` 固定へ変更する。
3. `StellaRecord` の停止対象 exe 名を構成整理後の実バイナリ名へ合わせる。
4. `StellaRecord` と `Alpheratz` に「ログイン時に起動」設定と初回選択 UI を追加する。
5. 最後に 3 アプリの NSIS と実行時コードを再監査し、レジストリ運用を再確認する。
