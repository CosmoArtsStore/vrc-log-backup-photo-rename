# Alpheratz 意図的な例外一覧

本一覧は「記述方式の統一」方針に対して、設計上の理由により意図的に残している例外点をまとめたものです。

## 1. 正規表現フォールバックの `expect`
- **ファイル:** `Alpheratz/src-tauri/src/scanner.rs`
- **該当箇所:** `Regex::new(r"^$").expect("fallback regex must be valid")`
- **理由:** フォールバック用の固定正規表現は不変であり、ここが失敗する場合はプロセス継続が不可能な異常状態。パニックよりも明確に落とすほうが安全。

## 2. ログ書き込み失敗時の `eprintln!`
- **ファイル:** `Alpheratz/src-tauri/src/utils.rs`
- **該当箇所:** `log_msg()` 内の `eprintln!`
- **理由:** ログファイルの書き込み自体が失敗している状況で `log_warn()` を使うと再帰ループになるため、標準エラー出力へ退避する「最後の手段」として残す。

## 3. 非Windows環境での `show_fatal_error` が無動作
- **ファイル:** `Alpheratz/src-tauri/src/main.rs`
- **該当箇所:** `#[cfg(not(target_os = "windows"))] fn show_fatal_error(...) { /* no-op */ }`
- **理由:** Windows 以外では MessageBox を使えないため、現在は無動作。CLI表示やダイアログAPIの導入までは例外として扱う。

**ループの流れ**

```
1. useEffect 実行
       ↓
2. startScan() 呼ばれる → scanStatus = "scanning"
       ↓
3. scan:completed 受信 → scanStatus = "completed"
       ↓
4. scanStatus が変わる → startScan が再生成される（useCallback）
       ↓
5. startScan が変わる → useEffect の依存配列が変わる
       ↓
6. useEffect が再実行される → また startScan() が呼ばれる 🔁
```

`scanStatus === "scanning"` の多重起動防止ガードは書いてありますが、**"completed" や "error" のときは素通りする**ので、スキャン完了のたびに再スキャンが走ります。

---

**修正の方向性**

`startScan` の依存配列から `scanStatus` を外すのが王道です。

```ts
const startScan = useCallback(async () => {
    setScanStatus(prev => {
        if (prev === "scanning") return prev; // ガードをsetter内で
        return "scanning";
    });
    // ...
}, [addToast]); // scanStatus を依存から除去
```

そして `useEffect` の中の `init()` は**マウント時1回だけ**動けばいいはずなので、依存配列を空にする or `startScan` を依存から外すかを検討する必要があります。
