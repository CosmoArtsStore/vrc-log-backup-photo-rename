# 開発・ビルドのガイドラインとトラブルシューティング

このドキュメントは本プロジェクトにおけるビルド時のルールと、よくあるエラーの解決手順をまとめたものです。
**これ以降の開発はこのドキュメントをベースに進めてください。**

## 🚨 絶対厳守のルール

1. **安易な設定書き換えによるビルドパス（ダーティハック）は絶対に行わない**
   - ビルドが通らないからといって、依存関係を無理やり消したり、`tauri.conf.json` などの設定ファイルを本来の仕様から外れる形で書き換えたりしてはならない。
   - 「本来通るべきビルドが通らない」場合、設定ではなく**環境やプロセスの状態**を疑うこと。

2. **3回以上似た原因でビルドエラーが連続した場合の対処**
   - むやみに再試行やハックによる解決を試みないこと。
   - **直ちに手を止め、原因を特定してユーザー（管理者）に報告・確認をとる**こと。

---

## 🛠️ よくあるビルドエラーと解決策

設定ファイルやソースコードに問題がないのに `npm run xxx:build` や `tauri build` が失敗する場合、以下の2パターンが原因であることが非常に多い。

### 1. アプリケーションが起動しっぱなしになっている（プロセスロック案件）
動作検証でアプリを起動したままテストを終え、バックグラウンドで `.exe` が動き続けているため、ビルド時の古いバイナリの上書きや削除ができずに `Permission Denied` や `os error 32` などで失敗するケース。

**【対処法】**
- タスクマネージャーを開き、該当のアプリケーションのタスクを終了する。
- `PhotoReNameApp.exe`, `CAS_LBTSetting.exe`, `OnsiteLogBackupTool.exe`, `LBTAppObserver.exe` が残っていないか確認する。
- PowerShell等のコマンドで強制終了する：
  ```powershell
  taskkill /F /IM PhotoReNameApp.exe
  taskkill /F /IM OnsiteLogBackupTool.exe
  ```

### 2. 開発プロセスによるファイルロック（PID停止案件）
開発中にViteの開発サーバー(`npm run dev`)やRustのプロセス(`cargo run`, `rust-analyzer` 等)が多発・ハングアップしてしまい、`target/` ディレクトリやポートをロックしているケース。

**【対処法】**
- 起動中のターミナルで `Ctrl + C` コマンドでプロセスを確実に終了できているか確認する。
- 見えないNode.jsやCargoのプロセスが残っている場合は、タスクマネージャーやリソースモニターから該当のプロセス(PID)を特定して終了する。
- 必要であれば、PCの再起動(操作はユーザーに依頼)や `cargo clean` を実施して `target/` ディレクトリ周りのロックを完全にクリアする。
