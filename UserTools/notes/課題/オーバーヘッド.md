以下、**「現状のオーバーヘッド」**を、主に ビルド/配布, 常駐時CPU・待機コスト, I/O/スキャンコスト, 運用・保守コスト の4軸で整理して分析します。
結論から言うと、いちばん大きいのは“純粋なCPU負荷”よりも、Tauriアプリ3本体制によるビルド/運用の重さと、Alpheratzの写真スキャン時I/Oの重さです。Polarisの常駐監視は常時負荷としては軽めですが、ポーリング設計ゆえの定常オーバーヘッドはあります。

1. 全体所見
先に結論
ビルド/配布オーバーヘッドが大きいです。
ルート package.json で Polaris / StellaRecord / Alpheratz の3アプリをワークスペース管理しつつ、それぞれを個別に tauri dev / tauri build しています。これは依存解決・ビルド・バンドル作業がアプリごとに分散する構造です。.

Polarisのランタイム負荷は“軽いがゼロではない”です。
常駐スレッドで 10秒ごとに全プロセス更新を行い、トレイメッセージループも 100ms ごとに wake-up しています。常駐監視アプリとしては妥当寄りですが、完全イベント駆動ではないため定常的な wake-up コストがあります。.

Alpheratzのスキャン処理が、実運用で最も重くなりやすいです。
再帰走査、既存DB照合、画像オープンによる向き判定、PNGメタデータ読取、必要に応じた StellaRecord DB 参照まで入っており、写真枚数に比例して I/O とデコード負荷が増える設計です。.

運用/保守オーバーヘッドも無視できません。
Polaris/Alpheratz ともに レジストリ上の InstallLocation 依存があり、Alpheratz はさらに StellaRecord の DB 参照までしているため、アプリ横断の結合が発生しています。設定・インストール状態・レジストリ差異に影響されやすい構造です。.

2. ビルド/配布オーバーヘッド
2-1. Tauriアプリが3本あり、各々が独立に重い
ルートでは3つのワークスペースを持ち、各アプリを個別の Tauri コマンドで起動/ビルドしています。
つまり、フロントエンド + Rust + Tauri bundling のパイプラインが3回発生する構成です。.

特に以下の点が重いです。

Polaris も Tauri を使っている。.

StellaRecord も Tauri + 複数 plugin + Rust依存を持つ。.

Alpheratz も Tauri + SQLite bundled + image 系依存を持つ。.

2-2. Polarisは機能に対して依存面がやや重い
Polarisの実装は、実態としては

単一インスタンス化

VRChatプロセス監視

ログ退避

トレイ常駐

が中心です。.

一方で依存は tauri, sysinfo, windows, chrono, tray-icon, image, winreg, dirs と広く、「バックグラウンド監視＋トレイ」用途に対してビルド面ではややヘビーです。これは機能上必要なものも多いですが、軽量ネイティブ常駐ツールとして見ると相対的には重い部類です。.

ここは 実装統一のメリット と ビルドコスト のトレードオフです。
現状は統一性を優先した構成、と読むのが自然です。

3. Polarisの常駐ランタイムオーバーヘッド
3-1. プロセス監視はポーリング方式
Polaris は監視スレッドでループし、VRChatがいなければ 10秒スリープ、見つけたら終了待ちに入ります。.

さらに、プロセス検出時には refresh_processes(ProcessesToUpdate::All, false) を呼んでいるため、全プロセス更新を伴います。これは10秒に1回なので常時重いわけではありませんが、イベント通知型よりは定常コストがある構成です。.

3-2. トレイループも100ms周期で起床
トレイ側は PeekMessageW を回しつつ、各ループ末尾で 100ms sleep を入れています。
これは busy loop を避けるための配慮として正しいですが、完全ブロッキング待機ではないため、常駐中は定期起床し続けます。.

3-3. ただし“致命的に重い”設計ではない
一方で、Polaris は VRChat を検知したら WaitForSingleObject(INFINITE) で対象プロセスの終了待ちに入るので、VRChat起動中に無駄な再ポーリングを続けるわけではありません。これは定常負荷を抑える方向に働いています。.

評価

常駐アプリとしての負荷: 低〜中

問題の本質: CPU負荷より設計上の wake-up / polling オーバーヘッド

すぐ問題化しそうか: 通常はしにくい

4. PolarisのI/Oオーバーヘッド
4-1. バックアップ時はログディレクトリ全走査
backup_logs() では VRChat のログディレクトリを read_dir で読み、output_log_*.txt を選別し、全件メタデータ確認したうえで必要なものだけコピーします。.

4-2. コピー回数は抑制されている
ただし、コピー条件は

output_log_*.txt

宛先が未存在、または

既存より source が大きい

に限定されています。
このため、毎回フルコピーになる設計ではなく、差分に近い抑制が入っています。】.

4-3. 他アプリをブロックしない配慮は良いが、I/O自体は発生する
Windows向けでは共有モード付きでソースを開き、VRChat側の読み書き削除を妨げない設計です。これは安全性の面で非常に良いです。
ただし当然ながら、バックアップ対象ファイルが多い・大きい時はI/O負荷はその分だけ増えます。.

評価

I/O負荷: イベント発生時のみ中程度

安全性: 高い

オーバーヘッドの性質: 常時ではなく“終了時バースト型”

5. Alpheratzのスキャン処理オーバーヘッド
ここが、実運用上もっとも重くなりやすい箇所です。

5-1. フルスキャン前提の流れが重い
do_scan() の主な流れは以下です。

写真ディレクトリ解決

DB接続

既存写真一覧取得

写真フォルダ再帰走査

既存データとの差分判定

必要な写真だけ解析

バッチ upsert

という流れです。】.

このうち、再帰走査 + 差分判定 + 個別ファイル解析が支配的コストになります。

5-2. 画像解析コストが乗る
写真ごとに状況次第で以下が走ります。

PNGメタデータ抽出。.

画像を開いて縦横判定。】.

特に image::open(path) は、ファイルI/O + デコードを伴うため、枚数が増えるほど重くなります。
また Rust 側依存にも image, image_hasher, rusqlite(bundled) があり、機能の厚みがそのままビルド・実行オーバーヘッドに反映されています。】.

5-3. StellaRecord DB への横断参照が追加コスト
Alpheratz は写真単体で world 情報が取れない場合、StellaRecord の stellarecord.db を開いて world_visits を参照します。
これは写真体験としては有益ですが、別アプリDBへの read-only 参照という運用依存とI/O依存を増やしています。】.

5-4. とはいえ、抑制策も入っている
全部を毎回フル再解析しているわけではなく、ScanRefreshKind により

Full

MetadataOnly

PathOnly

に分け、必要最低限に抑えようとしています。】.

さらに upsert はバッチ化されています。】.

評価

実行時オーバーヘッド: 高め

支配要因: ディスク走査・画像オープン・PNG読取・DB照合

改善余地: 大きい

6. フロントエンド側の“認知的オーバーヘッド”
6-1. Alpheratzの App.tsx は調整役としてかなり重い
App.tsx では多くの状態を一箇所で持ち、複数の custom hook と UI コンポーネントを束ねています。
設計としては hooks 分割がされているので破綻はしていませんが、画面挙動の中枢が1ファイルに集まりやすく、変更時の認知負荷は高めです。】.

6-2. FilterSidebar も局所的に複雑
FilterSidebar は props が多く、独自 state・日付UI・外クリック処理・絞り込みロジックを抱えています。
機能的には必要ですが、UI追加や不具合修正時の把握コストは高めです。】.

評価

CPUオーバーヘッドではなく、保守オーバーヘッド

将来的な変更コストの観点では無視しにくい

7. 運用オーバーヘッド・結合オーバーヘッド
7-1. レジストリ依存
Polaris も Alpheratz も、インストール先取得に HKCU レジストリを使っています。
つまり、単純なローカルアプリではなく、インストール状態に依存した運用構成です。】.

7-2. アプリ間結合
Polaris → 自身の install dir に archive / info.log を置く。】.

Alpheratz → StellaRecord install dir を引いて DB を読む。】.

このため、単一アプリで閉じない運用負担があります。
とくに障害解析時は「レジストリ」「インストール場所」「他アプリDBの状態」を横断して見る必要があり、保守・調査のオーバーヘッドが増えます。

8. 優先順位つき評価
優先度A: 一番効いているオーバーヘッド
Tauriアプリ3本体制によるビルド/配布負荷。.

Alpheratzスキャンのディスク/I/O/画像解析負荷。.

優先度B: 常駐時に効いているオーバーヘッド
Polarisの10秒ポーリング + 100msトレイ起床。】.

優先度C: 将来の変更コストとして効いているオーバーヘッド
フロントエンド中枢の集中化による認知負荷。】.

レジストリ/他アプリ依存による運用負荷。】.

9. 実務的な判断
現状のボトルネックを一言でいうと
常駐CPUが致命的に重いわけではありません。

ただし、構成全体としては“軽量”ではなく、特にスキャン時・ビルド時・運用時に重さが出る設計です。

ユーザー体感の主犯候補は Alpheratz のスキャン処理です。】.

常駐負荷の主犯候補は Polaris のポーリングですが、優先度は上2つより低いです。】.

結論
先に要点
Alpheratz のメモリを大きく食っている主因は、1つではなく3段構えです。

一覧用に全写真レコードを一括で WebView 側へ載せていること。
get_photos は LIMIT なしで全件返し、usePhotos がそれを丸ごと photos state に持っています。これは写真件数に比例して JS ヒープを増やす構造です。.

サムネイル生成・表示の流れが「カード単位の都度 invoke + 360x360 サムネイル」になっていること。
PhotoCard ごとに create_thumbnail を呼び、Rust 側では image::open → thumbnail(360, 360) を行っています。これはスクロール時のデコード負荷と、WebView が保持する decoded bitmap メモリの両方に効きます。】.

詳細モーダルで元画像をそのまま読み込んでいること。
PhotoModal は convertFileSrc(photo.photo_path) でオリジナル画像を直接表示しています。4Kや高解像度画像なら、1枚だけで数十MB級の decoded memory を持つことがあります。】.

この3つのうち、最優先で効くのは 1 と 2です。
150MB目標は現実的ですが、100MB目標は “WebView込み” だとかなり厳しいです。特に WebView2 のベースコストがあるため、コード改善だけで 100MB を安定達成するには、かなり攻めた削減が必要です。
私の見立てでは、まず 150MB台に落とす設計変更を優先するのが現実的です。

1. いちばん怪しい場所: 全件ロード
現状
Rust 側の get_photos は photos テーブルから photo_filename, photo_path, world_id, world_name, timestamp, memo, phash, orientation, is_favorite, match_source を 全件取得し、Vec<PhotoRecord> を返しています。ORDER BY timestamp DESC はありますが、LIMIT がありません。.

その結果をフロントの usePhotos が photos state にそのまま載せています。】.

Photo 型を見ると、一覧表示に本当に必要なのは一部なのに、現状は

photo_path

world_id

memo

phash

tags

match_source

まで全部 WebView 側へ持っています。】.

何が重いか
これは典型的に 「一覧表示に対してデータが太すぎる」 状態です。

特にメモリを押し上げる要因は次です。

photo_path はフルパス文字列なので長い。

memo は本文が長い可能性がある。

tags: string[] は配列オブジェクトを各行に持つ。

phash も一覧には不要。

全件を JS 側に持つので、さらに filteredPhotos, worldNameList, worldCounts, tagOptions などの派生計算が全件前提になります。】.

filteredPhotos 自体は参照配列なので破壊的ではないですが、元の photos が大きいこと自体が支配的です。
写真が数千件を超えると、JSオブジェクト + 文字列 + React管理コストでかなり膨らみます。

対応策
最優先対応
一覧用 DTO を別にするのが一番効きます。

たとえば一覧では、まずこれだけに絞ります。

type PhotoListItem = {
  photo_filename: string;
  thumbnail_path: string | null;
  world_name: string | null;
  timestamp: string;
  orientation: "portrait" | "landscape" | null;
  is_favorite: boolean;
  match_source: "metadata" | "title" | "stella_db" | null;
}
詳細表示時だけ別コマンドで取る形です。

get_photo_page(filters, offset, limit)
get_photo_detail(filename)
期待効果
JS ヒープをかなり削れます。

とくに memo, phash, world_id, tags, 長い photo_path を一覧から外すだけで、件数が多いほど効きます。

体感では、これが 最大の削減ポイントです。

目安
写真件数が多い環境では -20MB〜-60MB 以上 ありえます。
これは推定ですが、今の構造だと十分ありえる幅です。

2. 次に怪しい場所: サムネイルの作り方と持ち方
現状
PhotoCard はマウント時に create_thumbnail を呼び、その返り値パスを convertFileSrc で <img> に食わせています。】.

Rust 側の create_thumbnail は spawn_blocking で utils::create_thumbnail_file を呼びます。】.

その実体は、

image::open(path)

img.thumbnail(360, 360)

thumb.save(cache_path)

です。】.

何が重いか
2-1. 360x360 は WebView メモリ視点ではまだ大きい
画像のディスクサイズではなく、デコード後のピクセル数がメモリを決めます。
360x360 RGBA 相当だと 1枚あたり概算で約 0.5MB 前後です。
表示中・キャッシュ中のサムネイルが増えると、WebView 側で数十MB単位まで膨らみます。

2-2. カード単位の invoke はスクロール時の負荷源
react-window を使っているので DOM は絞られていますが、見える範囲のカードごとに invoke("create_thumbnail") を打つ構造です。】.

これにより、

スクロール時にサムネイル生成・取得が多発する。

画像 decode の一時メモリが発生する。

WebView が画像キャッシュを抱えやすい。

対応策
優先対応
サムネイルをスキャン時またはバックグラウンドで事前生成する。
つまりカード表示時に create_thumbnail を呼ばず、DBに thumbnail_path を保存して一覧はそれを見るだけにする。

サムネイルサイズを 360 → 160〜220 に落とす。
ここが重要で、JPEG品質を下げるだけでは decoded memory はあまり減りません。
メモリを下げるには画素数を下げる必要があります。

<img> に loading="lazy" と decoding="async" を付ける。
これは補助策ですが、WebView の負担を少し和らげます。

期待効果
一覧表示中の decoded bitmap が減る。

スクロール時のメモリスパイクが減る。

Rust 側の image::open 頻度も減る。

目安
-15MB〜-40MB は十分狙えます。
とくに 360px → 180px にすると、画素数は約 1/4 です。

3. モーダルの元画像直読みは確実に重い
現状
PhotoModal では <img src={convertFileSrc(photo.photo_path)} /> として、元画像を直接表示しています。】.

何が重いか
ブラウザ/WebView は表のために画像を decode します。
たとえば高解像度の VRChat スクショだと、圧縮ファイルは数MBでも decode 後は 20〜40MB 級になることがあります。
モーダルを閉じても、即時に完全解放される保証はありません。

対応策
優先対応
モーダル初期表示は 「プレビュー用中解像度画像」 にする。

元画像は

「原寸表示」ボタンを押した時だけ読み込む

もしくは外部ビューアで開く
に分ける。

具体案
preview_path を別途生成しておく。

モーダルはまず preview_path を表示。

ズーム要求時だけ original を読む。

期待効果
モーダルを開いた瞬間のメモリ増加を抑制。

-10MB〜-30MB の改善が期待できます。
特に高解像度写真では効きます。

4. 一覧以外で地味に効いているもの
4-1. 選択中写真をオブジェクトごと複製して持っている
usePhotoActions では selectedPhoto: Photo | null と photoHistory: Photo[] を持っています。】.

これは巨大な本体ではないですが、一覧側と同じ Photo オブジェクトを複数 state に持つ構造です。
さらに保存時には setPhotos(...map...) で配列全体を再構成しています。】.

対応策
selectedPhotoId だけを持つ。

photoHistory も filename[] にする。

詳細は photoMap または detail fetch で取る。

効果
単体では小さめですが、全体の state 複製を減らすのに効きます。

4-2. 全件前提の派生計算
worldNameList, worldCounts, tagOptions は photos 全件ベースです。】.

これは主に CPU 寄りですが、結果の配列や Set も増えるため、件数が多い環境では無視できません。

対応策
これらは DB 側で別集計 API にする。

少なくとも tagOptions は一覧ロードと分離した方が良いです。

5. スキャン中に増えるメモリ
現状
scanner::do_scan は

既存写真一覧を HashMap へ読み込み

found_files を Vec に溜める

candidate_files を Vec に作る

必要なら image::open(path) で向き判定

PNG メタデータも読む

という流れです。】.

何が重いか
これは常時メモリより スキャン時ピーク に効きます。

found_files と candidate_files でファイル一覧を二重に持つ。

existing_photos も HashMap で全件持つ。

image::open は画像デコードを伴う。

対応策
found_files と candidate_files をなるべく streaming 化する。

向き判定はスキャン時に必要最小限にし、別ジョブ化も検討。

image::open の同時実行数を増やしすぎない。

優先度

6. 150MB / 100MB の現実性
150MB目標
かなり現実的です。
次の3点をやれば届く可能性が高いです。

一覧を全件フルレコードで持たない。.

サムネイルを事前生成 + 180〜220px 化。】.

100MB目標
WebView込みだとかなり厳しいです。
理由は、アプリコード以前に

WebView ランタイムの固定費

React / CSS / 画像キャッシュ

Tauri 側の常駐コスト

があるためです。

達成するには、上記3点に加えてさらに

一覧ページング必須

詳細は完全 lazy load

一覧では photo_path すら持たない

サムネイルをもっと小さくする

モーダルで original を基本読まない

くらいまで踏み込む必要があります。

私の判断では、まず 150MB を安定達成する設計が先です。
