# コード調査スキル

## 目的

修正作業の前に問題箇所を正確に把握するための調査を行う。
**このスキルでコードの修正は行わない。調査・報告のみ。**

---

## 基本原則

- コードは一切修正しないこと
- 「おそらくこうだろう」という推測で報告しないこと。必ずファイルを開いて確認すること
- 存在しないファイル・行番号を報告しないこと
- 不明な点は「不明」「要確認」として報告すること
- 勝手に削除・統廃合・リネームの判断をしないこと

---

## 調査手順

### Step 1: 対象ファイルの特定

調査対象のキーワードで全ファイルをgrepする。

```bash
# 例：特定の文字列を含むファイルを全て列挙
grep -rn "キーワード" --include="*.rs" .
grep -rn "キーワード" --include="*.ts" .
grep -rn "キーワード" --include="*.json" .
grep -rn "キーワード" --include="*.nsi" .
```

ヒットしたファイルを全て列挙する。1件もヒットしない場合は「該当なし」と報告する。

### Step 2: 該当箇所の詳細確認

Step 1でヒットした各ファイルを開き、該当行の前後10行を確認する。

確認する内容：
- 正確なファイルパス（リポジトリルートからの相対パス）
- 行番号
- 該当コードの抜粋（前後3〜5行含む）
- そのコードが何をしているか（用途・意図）
- 問題の内容（なぜ問題なのか）

### Step 3: 影響範囲の確認

問題のあるコードが他の箇所から参照・呼び出されていないか確認する。

```bash
# 関数名・変数名・ファイル名で参照元を探す
grep -rn "関数名" .
```

### Step 4: 対応策の提示

問題箇所ごとに以下を提示する。ただし実施はしないこと。

- 推奨する修正方法
- 修正時の注意点
- 修正後に確認すべき事項

## キーワード別の調査観点

### パス・ディレクトリ系
```
LOCALAPPDATA / APPDATA / USERPROFILE
PathBuf / path() / join()
unwrap_or_default / unwrap_or_else / unwrap_or("")
```
→ ハードコードされたパスがないか。フォールバックが危険な形になっていないか。

### ログ系
```
println! / eprintln! / print!
error_info.log / crash.log / error.log
log_warn / log_err / log_msg
```
→ printlnが残っていないか。ログファイル名が統一されているか。

### NSIS系
```
WriteRegStr / DeleteRegKey
taskkill / ExecWait / nsExec
MUI_HEADERIMAGE / headerImage
Run（スタートアップ登録）
```
→ レジストリキーのパスが正しいか。taskkillの書き方が統一されているか。

### アプリ間連携系
```
pleiades.json / jewelBox.json / cosmos.json
stellarecord.db / alpheratz.db
polaris / stellarecord / alpheratz（小文字・大文字混在）
```
→ パスの表記が統一されているか。読み書きで食い違いがないか。

### Rust設定系
```
winreg / HKEY_CURRENT_USER
authors = / channel =
unwrap() / expect()
```
→ winregが依存に入っているか。authors・toolchainバージョンが統一されているか。

---

## 注意事項

**エージェントへの重要指示:**

調査中に明らかな問題を発見しても、その場で修正しないこと。必ず報告してから修正指示を待つこと。

「残っているから削除する」「古そうだから消す」「統合できそうだから統合する」といった自己判断による変更は禁止。全ての変更は明示的な指示に基づいて行うこと。

不明な点・判断に迷う点は全て「要確認」として列挙し、推測で処理しないこと。
