# STELLAProject コーディング規約

> 基準実装: `Alpheratz`
> 対象: STELLAProject 配下の全アプリ、全エージェント、補助スクリプト

## 1. 基本方針

- `Alpheratz` の責務分離と失敗時の挙動を正とする。
- Windows 向け VRChat 周辺ツールであることを前提に設計する。
- 業界標準のライブラリ、方式、ファイル形式、設計パターンで要件を満たせる場合は、独自実装より標準採用を優先する。
- 個人開発段階でも、将来のグループ開発、保守引き継ぎ、第三者レビューを前提に読みやすさと説明可能性を確保する。
- 例外は残してよいが、必ずコメントで「なぜ例外なのか」を明記する。
- 実装判断に迷ったら「データ保全」「復旧容易性」「調査容易性」を優先する。
- `Polaris` はユーザーデータと他アプリのログへ直接触れるため、STELLAProject 内でも特に慎重に扱う。
- `Polaris` では規約準拠より先に「変更理由と挙動を第三者が追えること」を重視し、理解しづらい構造を避ける。

## 1.1 標準採用優先

- 画像処理、ハッシュ、暗号、圧縮、シリアライズ、並列処理、認証、HTTP、DB アクセスなどの一般技術領域では、まず広く使われている標準ライブラリや定評ある実装を検討する。
- 「動くから」で独自実装を採用しない。少なくとも「既存標準で代替できるか」を確認してから判断する。
- 独自実装を採用する場合は、標準方式では満たせない要件、採用しない理由、保守上の注意点をコードまたは記録に残す。
- 既存の独自実装を見つけた場合も、必要に応じて標準方式への置き換え余地を検討する。
- 将来の共同開発者が「なぜその実装なのか」を外部知識なしで追えることを最低条件とする。

## 2. 構造方針

- 低レイヤーは I/O、DB、設定、ログに責務を限定する。
- アプリ境界では UI コマンドと内部ロジックを分離する。
- フロントエンドは表示、入力、状態管理に責務を限定し、永続化は Tauri command 経由で行う。
- 1 ファイル 1 主責務を原則とし、責務が増えたら hook / module / helper に分割する。
- ただし `Polaris` は運用挙動を読み解けることを優先し、無理に分割して追跡しづらくしない。

## 2.1 Polaris 追加方針

- `Polaris` の変更は「ユーザーデータや他アプリにどう触れるか」を先に文章で説明できる状態で行う。
- ログコピー、ファイル走査、プロセス監視の条件は、非 Rust 利用者でも読める粒度のコメントで残す。
- `Polaris` では過剰な抽象化、過剰な防御分岐、責務の細分化よりも、追跡しやすい直列な処理を優先する。
- 他アプリのファイルを開く処理は、相手側の読み書きや削除を妨げない共有方針を明記する。
- 失敗時に継続する場合も「何を諦めて何を継続するか」を説明できなければならない。

## 3. 命名規則

### 3.1 Rust

- `struct` / `enum` / `trait` / `type` は `UpperCamelCase`
- 関数、変数、モジュール、ファイルは `snake_case`
- `const` / `static` は `SCREAMING_SNAKE_CASE`
- 正規表現キャッシュは `RE_` プレフィックスで統一する

### 3.2 TypeScript / React

- コンポーネントは `UpperCamelCase`
- hook は `useXxx`
- 関数、変数、props、state は `camelCase`
- 型、interface、type alias は `UpperCamelCase`
- ファイル名はコンポーネントなら `UpperCamelCase.tsx`、hook と utility は `camelCase` ではなく既存実装に合わせて `useXxx.ts` / `index.ts` を使う

## 4. エラーハンドリング規約

- 通常の失敗は `Result` で返し、握り潰さない。
- `map_err(|e| e.to_string())` を禁止し、必ず文脈付きメッセージへ変換する。
- 継続可能な失敗は `WARN` ログを出して継続する。
- 継続不能な失敗は文脈付き `Err` として返す。
- `unwrap()` / `expect()` は不変条件が壊れた場合のみ許容し、理由コメントを必須とする。
- 戻り値を意図的に捨てる場合は、その理由をコメントで残す。

### 4.1 推奨パターン

```rust
fn scan_err<E: std::fmt::Display>(context: &str, err: E) -> String {
    format!("{}: {}", context, err)
}
```

```rust
match row {
    Ok(value) => Some(value),
    Err(err) => {
        crate::utils::log_warn(&format!("row decode failed: {}", err));
        None
    }
}
```

## 5. ログ規約

- ログは調査可能性のために残す。黙殺しない。
- 継続可能な失敗は `log_warn()`、継続不能な失敗は `log_err()` を使う。
- ログメッセージは文脈を含める。
- 個人情報、認証情報、秘匿値はログに出さない。
- ログ経路自体が壊れている場合の最後の退避先だけは例外として `stderr` を許容する。

## 6. 設定・フォールバック方針

- 設定読み込みは `fail-open` を許容するが、失敗時は `WARN` を必ず残す。
- 設定保存は `fail-fast` とし、呼び出し元へ `Err` を返す。
- パスや設定値の取得失敗を `Option` に落とす場合も、調査が必要な失敗はログを残す。

## 7. SQLite / 永続化規約

- SQL には必ずバインド変数を使い、文字列連結で値を埋め込まない。
- まとまった書き込みは transaction で囲う。
- 接続初期化時は `journal_mode = WAL`、`synchronous = NORMAL`、`foreign_keys = ON` を基本とする。
- 行デコード失敗を握り潰さず、継続可能なら `WARN` に落とす。
- DB エラーは操作内容が分かる文脈付きメッセージで返す。

## 8. Tauri / 境界設計

- command は UI と内部ロジックの境界とする。
- command 層では、下位層の失敗を UI に返せる `String` へ変換してよい。
- ただし command 層でも文脈無しの `to_string()` 変換は禁止する。
- 外部アプリ起動、URL オープン、Explorer 表示などの OS 依存操作は command 側に閉じ込める。

## 9. React / フロントエンド規約

- フロントエンドは表示と状態管理に集中し、DB やファイルの直接操作を禁止する。
- 書き込み系処理は `invoke()` を通し、成功時だけローカル state を更新する。
- 失敗時は toast などで利用者に伝える。
- `useEffect` は依存関係を厳密に扱い、再実行で意図しない副作用ループを起こさない。
- 計算ロジックは hook または helper に切り出し、UI コンポーネントに埋め込みすぎない。

## 10. コメント規約

- コメントは「何をしているか」ではなく「なぜそうしているか」を書く。
- 例外処理、Windows 固有事情、フォールバック、既知の制約には短い理由コメントを付ける。
- 古い事情を引きずった曖昧なコメントは残さず、現行実装に合わせて更新する。
- `Polaris` では、Rust を知らない保守者でも「いつ待機するか」「いつコピーするか」「なぜ安全か」が読めるコメントを優先する。

## 11. テスト・検証規約

- Rust 変更時は `cargo fmt` と `cargo clippy` を通す。
- 可能なら `cargo test` を実行する。
- フロントエンド変更時は最低でも型エラーとビルド可能性を確認する。
- バグ修正では「何が壊れていたか」「再発防止は何か」を説明できる状態にする。

## 12. レビュー基準

- 責務が明確か
- 失敗時の挙動が一貫しているか
- ログで追跡可能か
- UI から永続化への経路が安全か
- 例外が無言で混入していないか
- 将来の保守者が見て理由を説明できるか

## 13. 禁止事項

- 文脈無しエラー文字列化
- 理由無し `unwrap()` / `expect()`
- 理由無し `let _ = ...`
- 失敗の黙殺
- ユーザー入力を含む SQL 文字列連結
- フロントエンドからの直接ファイル操作、直接 DB 操作
- ハードコードされた環境依存パスの恒久実装

## 14. 例外の扱い

- 例外は許可制ではなく説明責任制とする。
- 例外を入れる場合は、コードコメントで「意図的」「代替案」「残す理由」を短く書く。
- 規約違反ではなく「説明済みの設計上の例外」にできるかを常に判断する。
